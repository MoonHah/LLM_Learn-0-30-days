## 一、项目目标

请帮我构建一个通用的企业级 Agentic RAG 系统模板。

系统需要具备以下能力：

1. 用户输入一个复杂业务问题。
2. Agent 判断问题类型和所需信息。
3. Agent 动态决定是否需要检索知识库。
4. Agent 可以根据任务需要调用不同工具，例如：
   * 向量检索工具
   * SQL 查询工具
   * 外部 API 工具
   * 文档解析工具
   * 业务规则校验工具
   * 风险检查工具
5. Agent 支持多跳推理，而不是只检索一次。
6. 如果检索结果不足，Agent 能够自动重试、改写查询或切换工具。
7. Agent 需要具备明确的终止条件，避免无限循环。
8. 系统需要记录完整执行链路，方便审计和问题排查。
9. 高风险操作必须进入人工确认流程。
10. 最终输出答案时，需要说明依据、置信度和不确定性。

## 四、系统应采用 Agentic RAG 架构

系统不能是传统固定流程 RAG：

用户问题 → 检索 → 生成答案

而应该是 Agentic RAG：

用户问题
→ Agent 分析任务
→ 判断是否需要检索
→ 选择工具
→ 执行工具
→ 观察结果
→ 判断信息是否充分
→ 如果不足，则改写查询 / 换工具 / 重试
→ 综合结果
→ 输出最终答案

请体现 ReAct 思想：

Thought → Action → Observation → Thought → Action → Observation → Final Answer

但是在代码实现中，不需要暴露模型的私有思考过程给最终用户。
可以用结构化日志记录执行步骤，例如：

* step_id
* selected_tool
* tool_input
* tool_output_summary
* decision_reason_summary
* error
* timestamp

## 五、LangGraph 设计要求

请使用 LangGraph 进行流程编排。

至少包含以下节点：

1. `classify_task`
   * 判断用户问题属于哪类业务任务。
   * 例如：知识问答、数据分析、合规检查、文档总结、多跳检索等。
2. `plan_next_step`
   * 判断下一步应该调用哪个工具。
   * 判断当前信息是否足够。
   * 判断是否需要继续检索。
3. `retrieve_knowledge`
   * 执行知识库检索。
   * 支持 query rewrite。
   * 支持 top_k 参数。
   * 支持检索结果为空时的 fallback。
4. `query_structured_data`
   * 执行结构化数据查询。
   * SQL 必须参数化。
   * 只能执行只读查询。
   * 禁止执行 DELETE、UPDATE、INSERT、DROP、ALTER 等危险语句。
5. `call_external_api`
   * 模拟调用外部业务 API。
   * 必须包含超时、重试和错误返回。
6. `evaluate_evidence`
   * 判断当前证据是否足够。
   * 判断是否需要继续检索。
   * 判断置信度是否足够。
7. `risk_check`
   * 检查是否涉及高风险操作。
   * 如果涉及金融、医疗、法律、人事、资金、隐私等内容，需要标记为高风险。
   * 高风险内容必须进入人工确认或输出免责声明。
8. `generate_answer`
   * 基于已有证据生成最终答案。
   * 答案需要包含：
     * 结论
     * 依据
     * 不确定性
     * 建议下一步
     * 是否需要人工确认
9. `handle_error`
   * 统一处理工具失败、超时、检索为空、SQL 异常等问题。

## 六、Agent State 设计要求

请使用 Pydantic 或 TypedDict 定义 Agent 状态。

State 至少包含：

* `user_query`: 用户原始问题
* `task_type`: 任务类型
* `current_step`: 当前步骤编号
* `max_steps`: 最大步骤数，默认 10
* `messages`: 对话消息
* `evidence`: 当前已获得的证据列表
* `tool_calls`: 工具调用记录
* `errors`: 错误记录
* `risk_level`: 风险等级，例如 low / medium / high
* `need_human_review`: 是否需要人工确认
* `confidence_score`: 置信度
* `is_finished`: 是否完成
* `final_answer`: 最终答案
* `history_summary`: 历史摘要，用于 Token 压缩
* `loop_signatures`: 用于检测是否陷入循环

## 七、工具设计要求

所有工具必须遵循清晰的工具描述规范：

1. 工具名必须清晰表达用途。
2. 工具 docstring 必须说明工具能做什么。

7. 工具调用必须可记录日志。
8. 高风险工具必须具备权限控制或人工确认。

5. `summarize_evidence(evidence: list) -> dict`
   * 对当前证据进行摘要压缩，避免 Token 无限增长。

## 八、生产环境硬性要求

请在代码中体现以下企业级要求：

### 1. 终止条件

必须设置最大迭代次数，例如：

* `max_steps = 10`
* 如果 `is_finished = True`，立即停止。
* 如果检测到循环，立即停止。
* 如果连续多次工具失败，进入错误处理。
* 如果置信度达到阈值，可以提前结束。

### 2. 错误处理

每个工具都必须包含：

* try-except
* timeout
* retry
* fallback
* 错误日志
* 结构化错误返回

不要让工具异常直接导致整个 Agent 崩溃。

### 3. Token 控制

不能让历史记录无限增长。

需要实现：

* `MAX_HISTORY_TOKENS`
* 历史摘要压缩
* 只保留关键 Observation
* 最终回答只使用必要证据

### 4. 安全边界

必须包含：

* SQL 参数化查询
* SQL 只读权限
* 工具白名单
* 禁止执行任意代码
* 禁止访问未授权文件
* 禁止泄露敏感信息
* 对 PII 信息进行过滤或脱敏
* 高风险操作需要人工确认

### 5. 审计追溯

每次工具调用必须记录：

* 请求 ID
* 用户问题
* 当前步骤
* 工具名称
* 工具输入
* 工具输出摘要
* 是否成功
* 错误信息
* 时间戳
* 风险等级

### 6. 风险控制

如果涉及以下场景，必须标记为高风险：

* 金融投资建议
* 医疗诊断
* 法律结论
* 人事决策
* 资金操作
* 隐私数据
* 删除、修改、提交、下单等不可逆操作

高风险任务不能直接自动执行，必须进入人工确认流程。

### 7. 性能优化

请设计并实现：

* 并行检索
* 结果缓存
* 提前终止
* 重复 query 去重
* 工具调用限流
* 可配置 top_k
* 可配置 max_steps

## 九、输出答案要求

最终 Agent 输出答案时，需要使用结构化格式：

```text
结论：
...

依据：
1. ...
2. ...

置信度：
高 / 中 / 低

不确定性：
...

风险提示：
...

是否需要人工确认：
是 / 否

建议下一步：
...
```

如果证据不足，Agent 不能强行回答。
应该输出：

```text
当前证据不足，无法给出可靠结论。
建议补充以下信息：
1. ...
2. ...
```
